# 蒸馏链综合改造实施总路线

## 1. 结论

`M7-交易风格 Archetype 与 Pattern 识别改进方案` 的方向是对的，命中了“统计特征已经有，但没有沉淀成可辨识 archetype”的核心问题。

`reflection-问题剖析与改进方案` 也成立，但它实际上包含两类不同问题：

- 运行稳定性问题：Pi runtime abort、timeout、模型/鉴权不可用、结构化输出不稳
- 内容质量问题：没有 archetype taxonomy、schema/quality gate 对泛化输出处理过硬、数据稀疏地址没有合法成功态

因此，这两份方案不能并排同时重写。正确做法是按“先稳运行壳，再补 archetype 数据层，最后升级 reflection 内容契约”的顺序实施。

## 2. 推荐顺序

### 阶段 A：Reflection 稳定性底座

目标：先降低 fallback 率，建立失败类型可观测性，避免后续改 prompt/schema 时问题被 runtime 噪声掩盖。

本阶段只处理：

- failure taxonomy
- timeout 策略
- model fallback
- salvage/raw_text 保留
- validator 的“分级接受”框架

本阶段不要做：

- archetype taxonomy 注入
- profile/schema 大扩容
- trade_pairing 大改

理由：

- 现在很多失败不是“抽不出风格”，而是 runtime 直接 abort
- 如果 runtime 不稳，后续新增 archetype 字段只会增加输出复杂度和失败面

### 阶段 B：Archetype 与 Pattern 数据层

目标：把现有 trade-level / stat-level 特征真正交叉成 pattern，形成可复用的中间语义层。

本阶段只处理：

- `trade_pairing.py` 扩展交易级与统计级特征
- 新增 `archetype.py`
- `service.py` 中 `derived_stats` / `compact_input` 扩展
- `extractors.py` fallback 逻辑改为优先消费 archetype 输出

本阶段不要做：

- 直接让 LLM 自己发明 taxonomy
- 直接扩大 reflection schema 而不先沉淀稳定中间层

理由：

- archetype classifier 是给 reflection 和 fallback 共用的“硬骨架”
- 如果不先把 pattern 算出来，prompt 再强也容易变成空泛 narrative

### 阶段 C：Reflection 内容契约升级

目标：让 LLM 基于 archetype taxonomy 和 pattern evidence 输出更有辨识度的风格标签，同时把“无明显模式”纳入合法成功态。

本阶段处理：

- `DEFAULT_EXTRACTION_PROMPT` 升级
- reflection output schema 扩容
- generic gate 从“硬拒绝”改为“分级接受”
- `insufficient_signal / no_pattern` 合法输出通道
- memory/backtest/quality gate 的状态联动

理由：

- 只有在阶段 B 完成后，prompt/schema 才有可靠数据可以绑定
- 否则 schema 会被扩容，但底层并没有足够证据承接

### 阶段 D：下游对齐与验收

目标：让 skill 编译、memory 和记忆复用、QA 结论都认识新的 archetype 与状态机。

本阶段处理：

- skill 编译文案与 manifest 映射
- QA 报告字段
- memory 记忆策略
- 回归样本集和验收基线

## 3. 依赖关系

### 必须串行

- 阶段 A -> 阶段 C
- 阶段 B -> 阶段 C
- 阶段 C -> 阶段 D

### 可以并行

- 阶段 A 和 阶段 B 可以并行开工，但合并前必须先完成阶段 A 的失败分类和 validator 通道定义

原因：

- 阶段 B 需要知道最终有哪些合法状态会进入 reflection 与 fallback
- 阶段 C 需要消费阶段 B 的 archetype 输出

## 4. 模块边界

### Reflection 模块负责

- runtime 调用链
- model 选择与超时
- raw output -> normalized output
- schema 校验
- reflection 状态分类

### Archetype 模块负责

- 交易级 pattern 证据沉淀
- 统计级 archetype 判定
- 行为模式证据摘要
- fallback profile 的风格骨架

### 耦合点

- `compact_input.derived_stats`
- `profile.style_label` / `profile.trading_archetype`
- `review.status`
- `strategy_quality` 与记忆门禁

## 5. 建议交付拆分

### 交付 1：Reflection 稳定性补丁包

- 失败分类
- timeout/model fallback
- validator 分级接受骨架

### 交付 2：Archetype 数据与分类器

- `trade_pairing` 扩展
- `archetype.py`
- fallback extractor 升级

### 交付 3：Reflection taxonomy/schema 升级

- prompt
- schema
- memory / review status 对齐

### 交付 4：下游验收包

- QA 模板
- 回归样本
- skill 输出检查项

## 6. 不建议的实施方式

- 不要先改 prompt/schema，再去补 archetype 数据
- 不要把 runtime 稳定性问题和内容质量问题混成一个任务
- 不要简单放松 generic 校验而不增加 `no_pattern` 合法态
- 不要把 archetype classifier 设计成只能给 LLM 用，fallback 也必须消费它

## 7. 最终建议

落地顺序定为：

1. Reflection 稳定性与状态机
2. Archetype 与 Pattern 数据层
3. Reflection taxonomy/schema 升级
4. Memory、QA、Skill 输出联调

这是风险最小、返工最少、最适合直接排期开发的顺序。
