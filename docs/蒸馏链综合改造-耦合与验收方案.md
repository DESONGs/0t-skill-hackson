# 蒸馏链综合改造-耦合与验收方案

## 1. 用途

本文件用于把 Archetype 模块和 Reflection 模块的交界面、开发边界、风险和验收口径一次性说清楚，避免后续并行开发时互相覆盖。

## 2. 耦合图

```text
wallet_profile / token_profiles / signals
    -> trade_pairing + statistics
    -> archetype classifier
    -> compact_input / derived_stats
    -> reflection runtime
    -> validated profile / strategy / review
    -> skill compiler / QA / memory
```

真正的耦合点只有四个：

- `trade_pairing/statistics` 输出字段
- `compact_input.derived_stats`
- reflection output schema
- `review.status / strategy_quality` 状态机

## 3. 开发边界

### Archetype 组可独立推进的部分

- `trade_pairing.py`
- `archetype.py`
- `service.py` 中 `derived_stats` 扩展
- `extractors.py` fallback 改造

### Reflection 组可独立推进的部分

- TS runtime 稳定性
- Python 失败分类
- validator 分级接受骨架
- observability

### 必须联合评审的部分

- schema 字段新增
- `review.status` 新状态
- memory 过滤条件
- QA 报告字段

## 4. 推荐排期

### Sprint 1

- Reflection R1/R2
- Archetype A1/A2

交付标准：

- runtime 失败类型可见
- archetype 所需底层特征已可计算

### Sprint 2

- Archetype A3/A4/A5
- Reflection R3

交付标准：

- fallback extractor 可消费 archetype
- validator 具备 `insufficient_signal` 合法态

### Sprint 3

- Reflection R4/R5
- Compiler / QA / Memory 对齐

交付标准：

- reflection schema 对接 archetype
- QA 与 skill 输出认识新字段

## 5. 验收矩阵

### 类型一：高信号地址

预期：

- 不走 fallback 或 fallback 也能输出明确 archetype
- 主 archetype 稳定
- behavioral pattern 可解释

### 类型二：中信号地址

预期：

- 可以生成 `generate_with_low_confidence`
- archetype 可能单一但不应机械拼接
- 不因个别字段空缺直接失败

### 类型三：低信号地址

预期：

- 输出 `insufficient_signal` 或 `no_stable_archetype`
- 不生成假的强模式
- 不应被误计为 runtime failure

### 类型四：runtime 异常场景

预期：

- 明确标记为 `runtime_failed`
- 失败类型清晰
- raw_text / request / model 信息可追踪

## 6. 核心指标

### 稳定性指标

- reflection success rate
- fallback rate
- runtime abort rate
- timeout rate

### 内容质量指标

- 非机械 style label 占比
- archetype 覆盖率
- `insufficient_signal` 占比
- behavioral pattern evidence 完整率

### 业务指标

- 策略师主观评分
- skill 是否能快速说清“这是哪类交易者”
- 新地址上的人工复核通过率

## 7. 验收样本建议

至少准备四组地址：

- 高频 meme scalper
- microcap day trader
- hold/diamond hands
- 低信号或无稳定模式地址

每组都要跑：

- reflection 原始结果
- fallback 结果
- final skill 输出
- QA 报告

## 8. 回归检查清单

- `trade_pairing` 扩展后，旧地址的配对覆盖率不明显退化
- fallback profile 不再主要依赖 `risk_appetite + execution_tempo`
- generic 输出不会直接把整轮打入 fallback
- `insufficient_signal` 不再被当成失败
- public/private 结果字段兼容

## 9. 决策结论

两个模块是“相对独立、接口耦合”的关系。

正确的执行方式不是：

- 一次性大改所有文件

而是：

1. Reflection 先稳住运行与状态机
2. Archetype 再补 pattern 中间层
3. Reflection 最后接入 taxonomy/schema
4. QA、memory、compiler 统一收口

只要按这个顺序推进，改动面虽然大，但仍然是可控的。
