# M3 - 市场环境上下文模块（上下文防爆重点）

## 模块定位

对应 harness 中的「市场环境上下文 (Market Context)」。这是 harness 明确标注为**蒸馏关键**的部分——单看交易记录没用，必须知道交易时市场在发生什么。

**同时，这也是最容易导致 Pi Agent 上下文爆炸的部分。**

---

## 现有代码能力评估

### 已实现 ✓

| 能力 | 代码位置 | 评价 |
|------|---------|------|
| Token 价格快照 | `_compact_token_snapshot → price_usd` | **不足**。只有当前价格，没有交易发生时的历史价格 |
| 流动性快照 | `_compact_token_snapshot → liquidity_usd` | **基础**。只有当前流动性 |
| 24h 交易量 | `_compact_token_snapshot → volume_24h_usd` | **基础**。只有当前 24h 量 |
| 风险等级 | `_compact_token_snapshot → risk_level, flags` | **好**。提供了风险标签 |
| 信号过滤 | `_filter_signals` | **好**。按 focus token 和 preferred symbol 过滤，Top-5 截断 |
| Top-K 截断 | `_preprocess_wallet_data` | **好**。holdings 5, activity 8, tokens 4, signals 5 |

### 缺失 ✗

| harness 需求 | 描述 | 严重程度 |
|-------------|------|---------|
| **价格动量数据** | 买入前 1h/24h 的价格走势，判断「追涨」vs「抄底」 | **高** — 核心蒸馏信号 |
| **波动率状态** | 交易发生在极端波动期还是横盘期 | **高** — 决定止损模型 |
| **链上热度数据** | 当时 Token 的全链交易量、社交媒体热度 | 中 — AVE 信号接口部分覆盖 |
| **关联资产走势** | 买入 A 时 BTC/ETH 的状态 | 中 — 可选项，但有助于判断大盘环境 |
| **历史时间点市场状态** | 交易发生时的市场快照（而非当前快照） | **高** — 否则蒸馏是"事后诸葛" |

## 上下文爆炸风险分析

### 当前状况

Pi 反射模式 `maxTokens: 3000`（output），input 无显式限制。当前 `compact_input` 的 JSON 大小约 2-4KB（tokens 少时），但如果加入市场环境数据，可能膨胀到：

| 数据类型 | 无控制大小 | 受控大小 |
|---------|----------|---------|
| 当前 compact_input | 2-4 KB | — |
| 每 Token 的 1h K线（60 点） | 4 KB × 4 tokens = 16 KB | 不可接受 |
| 每 Token 的摘要指标 | 0.2 KB × 4 = 0.8 KB | ✓ |
| 大盘 BTC/ETH 状态 | 0.3 KB | ✓ |
| 波动率指标 | 0.1 KB × 4 = 0.4 KB | ✓ |
| 全部信号详情 | 不定 | 截断到 5 条 ≈ 1 KB |

**原则**：LLM 只接收**预计算的摘要指标**，不接收原始时序数据。

### 上下文预算

```
总预算: ≤ 6 KB 的 compact JSON（约 2000 tokens）

分配:
- 钱包基础信息 + 持仓:    ~1.5 KB （现有，保持不变）
- 交易活动样本:           ~1.5 KB （现有 8 条，保持不变）
- M2 统计特征:            ~0.3 KB （新增，仅汇总数字）
- M3 市场上下文摘要:       ~1.2 KB （新增，重点控制项）
- M4 过滤后信号:          ~1.0 KB （现有 5 条）
- derived_stats:          ~0.5 KB （现有 + M2 扩展）
```

## 核心设计：两层架构

### 第一层：预计算层（Python 侧，不限 Token）

在 Python 端（非 LLM 端）完成所有时序计算，产出摘要指标。

```python
@dataclass
class TokenMarketContext:
    symbol: str
    token_address: str
    
    # 价格动量（预计算）
    price_now: float
    price_1h_ago: float | None
    price_24h_ago: float | None
    price_change_1h_pct: float | None   # 正=上涨
    price_change_24h_pct: float | None
    momentum_label: str                  # "pumping" | "dumping" | "ranging" | "recovering"
    
    # 波动率（预计算）
    volatility_24h_pct: float | None     # 24h 价格标准差 / 均价 × 100
    atr_24h_usd: float | None            # Average True Range
    volatility_regime: str               # "extreme" | "high" | "normal" | "low"
    
    # 流动性与热度
    liquidity_usd: float | None
    volume_24h_usd: float | None
    volume_to_liquidity_ratio: float | None  # >1 = 热门
    
    # 交易发生时的市场状态（回溯）
    trade_time_momentum: str | None      # 该交易员买入时的价格趋势


@dataclass
class MacroContext:
    btc_price_usd: float | None
    btc_24h_change_pct: float | None
    eth_price_usd: float | None
    eth_24h_change_pct: float | None
    market_regime: str                   # "risk_on" | "risk_off" | "neutral"
```

### 第二层：压缩层（注入 LLM 的 JSON）

将第一层结果压成**扁平化 JSON**：

```json
{
  "market_context": {
    "macro": {
      "btc_24h_pct": -2.3,
      "eth_24h_pct": -3.1,
      "regime": "risk_off"
    },
    "focus_token_context": [
      {
        "symbol": "PEPE",
        "price_1h_pct": 12.5,
        "price_24h_pct": 45.0,
        "momentum": "pumping",
        "volatility_regime": "extreme",
        "vol_liq_ratio": 3.2
      },
      {
        "symbol": "ARB",
        "price_1h_pct": -1.2,
        "price_24h_pct": 5.3,
        "momentum": "ranging",
        "volatility_regime": "normal",
        "vol_liq_ratio": 0.8
      }
    ]
  }
}
```

**每个 token 约 0.15 KB，4 个 token + macro ≈ 0.9 KB**，完全在预算内。

## 数据获取方式

### 方案 A：利用现有 `inspect_market` API（推荐 MVP）

`AveRestProvider.inspect_market` 已经调用了 `kline-pair`（K线）和 `pair`（交易对），可以从中提取：

```python
def extract_market_context(market_data: dict) -> TokenMarketContext:
    ohlcv = market_data.get("ohlcv", [])
    if len(ohlcv) >= 2:
        latest = ohlcv[-1]
        hour_ago = ohlcv[-2] if len(ohlcv) >= 2 else None
        day_ago = ohlcv[0] if len(ohlcv) >= 24 else None
        # 计算 price_change、volatility 等
    ...
```

**优点**：不需新增 API，复用已有能力。
**缺点**：需要额外一轮 `inspect_market` 调用（当前蒸馏流程未调用此接口）。

### 方案 B：批量 Macro 快照（BTC/ETH）

新增一个轻量函数获取大盘状态：

```python
def fetch_macro_context(provider) -> MacroContext:
    btc = provider.inspect_token(InspectTokenRequest(
        token_ref=TokenReference(identifier="ethereum:btc", chain="ethereum", symbol="BTC")
    ))
    eth = provider.inspect_token(InspectTokenRequest(
        token_ref=TokenReference(identifier="ethereum:eth", chain="ethereum", symbol="ETH")
    ))
    return MacroContext(
        btc_price_usd=btc["market_snapshot"]["price_usd"],
        btc_24h_change_pct=...,
        ...
    )
```

这个调用**可与 M1 的 token_profiles 并行执行**。

## 与现有代码的结合点

### 注入位置

在 `_preprocess_wallet_data` 输出中新增 `market_context` 字段：

```python
def _preprocess_wallet_data(..., market_contexts=None, macro_context=None):
    # ... 现有逻辑不变 ...
    
    result = {
        # ... 现有字段 ...
        "market_context": {
            "macro": macro_context.to_compact() if macro_context else None,
            "focus_token_context": [
                ctx.to_compact() for ctx in (market_contexts or [])
            ],
        },
    }
    return result
```

### 现有代码中好的设计（保留）

- `_compact_token_snapshot` 的压缩思路正确，只是字段不够
- `_filter_signals` 的 focus token 过滤策略正确
- Top-K 截断策略正确，不应放松

### 现有代码中需改进的

- `token_snapshots` 当前只有 4 个静态字段（price, liquidity, volume, risk），需要扩展为包含动量和波动率的版本
- 但**不应**替换现有字段，而是在 `market_context` 中**新增**，保持向后兼容

## 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 原始K线是否给LLM | **否** | 60个OHLCV数据点约4KB/token，4个token就是16KB，会挤压其他上下文空间 |
| 预计算指标数量 | 每token 6个指标 | 对LLM来说足以判断"追涨/抄底/横盘"，不需要更细粒度 |
| 历史时间点回溯 | **MVP阶段不做** | 获取交易发生时的历史市场状态需要时间点K线查询，API成本高。先用当前快照，后续可增量 |
| BTC/ETH macro | **做** | 仅 2 次 API 调用，信息密度高，LLM理解大盘环境的关键输入 |

---

## 风险与注意事项

1. **上下文预算是硬约束**：任何新增字段都必须经过 `json.dumps` 后计算字节数，确保总量 ≤ 6KB
2. **波动率计算依赖 K 线质量**：AVE 的 `kline-pair` 在某些小 token 上可能返回稀疏数据，需要 graceful degradation
3. **不要引入「动态上下文」**：所有 market_context 字段必须是**静态快照值**（数字/字符串），不允许嵌套列表或时序数组
4. **momentum_label 的分类标准需要与交易员对齐**：建议用简单百分比阈值（>10% = pumping, <-10% = dumping, 其余 = ranging），可后续调整
