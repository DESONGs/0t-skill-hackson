# 策略蒸馏模块化总结指引

## 一、全局视图：Harness 文档 vs 现有代码的交叉辩证

### 1.1 现有代码做得好的部分

| 能力 | 评价 | 理由 |
|------|------|------|
| AVE Provider 适配层 | **优秀** | `AveRestProvider` 对 AVE CLI 的 JSON 归一化非常健壮，处理了大量格式不一致问题。Mock/Live 双模式切换干净 |
| Top-K 截断策略 | **优秀** | holdings 5、activity 8、tokens 4、signals 5 的截断是防止上下文爆炸的第一道防线，设计合理 |
| Pi Reflection 双路径 | **优秀** | mock → live → fallback 三级降级，artifact 全量持久化，可追溯性强 |
| Skill 包编译器 | **优秀** | `SkillPackageCompiler` 生成完整的 SKILL.md + manifest + actions + interface + primary.py，契约验证完善 |
| Candidate 生命周期 | **优秀** | compile → validate → promote → smoke 的标准流水线，有 QA 检查点 |
| Focus Token 选择 | **好** | `_pick_focus_tokens` 按交易频次 + 持仓金额加权排序，fallback 到 holdings |
| 时间窗口 / 爆发检测 | **好** | `_active_window_labels` + `_burst_profile` 对交易时段和密集度有合理判断 |

### 1.2 现有代码的关键缺口

| 缺口 | 严重程度 | 对应模块 | 说明 |
|------|---------|---------|------|
| **数据采集串行** | P0 | M1 | 4 个 token 的 inspect_token 串行执行，延迟约 40-60s，应并行 |
| **缺少买卖配对** | P0 | M2 | 无法计算胜率/盈亏比/持仓周期，这是蒸馏的统计基础 |
| **缺少市场动量/波动率** | P0 | M3 | 只有当前价格快照，没有价格变化率、波动率状态 |
| **Extraction Prompt 太笼统** | P0 | M5 | 不引导 LLM 生成具体的入场/出场条件 |
| **primary.py 逻辑过简** | P0 | M5 | 只有 market_bias + risk_appetite 两个 if-else，不符合 harness 描述的触发条件 |
| **缺少执行适配层** | P0 | M5/M6 | 当前 skill 只能输出 plan，不能把交易计划映射到真实签名/模拟/广播链路 |
| **缺少入场因子蒸馏** | P1 | M4 | 没有从盈利交易中提取共性入场条件 |
| **缺少回测验证** | P1 | M6 | smoke test 只验证脚本能跑，不验证策略质量 |
| **缺少止损/止盈模型** | P1 | M5 | harness 要求阶梯止盈、ATR 动态止损，代码完全没有 |
| **历史交易只取一页** | P1 | M1 | harness 要求全量交易流水，当前只拉最近一页 |
| **缺少 LP 锁定 / Holder 集中度过滤** | P2 | M4 | harness 明确要求，AVE API 部分支持但未集成到过滤链路 |
| **缺少大盘(BTC/ETH)宏观状态** | P2 | M3 | 判断"风险偏好开/关"的重要输入 |

### 1.3 Harness 文档中需务实调整的部分

| 文档描述 | 调整建议 | 理由 |
|---------|---------|------|
| "全量交易流水" | 改为**分页拉取最近 5 页** | 全量可能数千笔，AVE API 有限流，且大部分历史对近期策略蒸馏价值低 |
| "RSI < 30 AND WhaleInflow > $50k" | MVP 阶段使用**价格变化率 + 成交量比值**替代 RSI | RSI 需要 14 期 K 线数据，依赖 `inspect_market` 额外调用 |
| "交互协议分布（Uniswap/Curve）" | **暂时搁置** | harness 已标注"若无暂时忽略"，AVE 当前 activity 数据不含 DEX 来源 |
| "社交媒体热度" | **暂时搁置** | AVE signals 接口有部分覆盖，但无法保证实时性 |
| "回归分析（相关性分析）" | MVP 简化为**频率统计 + 规则匹配** | scipy/sklearn 回归分析在 <20 个样本上没有统计显著性 |
| "置信度评分决定头寸大小" | **保留，但标注为「建议值」** | AI 建议头寸不等于自动执行，需人工确认 |
| "自动交易策略" | 拆成**AVE-only 策略蒸馏 + onchainos CLI 执行适配层** | 避免双数据路径；读和回测仍只走 AVE，执行单独走 CLI |

## 三、并行执行策略

### 3.1 M1 内部并行

```
inspect_wallet ──┬──→ inspect_token × 4 (ThreadPool)  ──→ 汇总
                 ├──→ review_signals                   ──→ 汇总
                 └──→ inspect_market × 4 (ThreadPool)  ──→ 汇总 (M3 数据)
```

**预计延迟**：串行 ~60s → 并行 ~15s

### 3.2 M2 / M3 / M4(部分) 并行

M1 完成后，以下三个任务**完全独立**，可同时启动：

| 任务 | 输入 | 预计耗时 |
|------|------|---------|
| M2: 交易配对 + 统计 | `full_activity_history` | ~1s (纯 Python 计算) |
| M3: 市场上下文压缩 | `token_profiles` + `market_data` | ~1s (纯 Python 计算) |
| M4: 风控过滤器构建 | `token_profiles` + `signals` | ~1s (纯 Python 计算) |

### 3.3 关键路径

```
M1(15s) → [M2+M3+M4并行](1s) → M4入场因子(1s) → M5(5-10s,LLM调用) → M6(1s)
总计预计: 23-28s （当前: 60s+，改进约 2-3x）
```

## 四、数据预处理链路精简方案

### 4.1 两层数据架构

```
┌────────────────────────────────────────────────────┐
│                  完整数据层 (Python 侧)               │
│  - full_activity_history (分页拉取，可能 100+ 条)      │
│  - 完整 token_profiles (含 holder_snapshot)           │
│  - 完整 market_data (含 OHLCV K线)                   │
│  - 完整 completed_trades (配对结果)                    │
│                                                      │
│  用途：M2 统计计算、M4 入场因子分析、M6 回测           │
│  存储：artifacts/*.raw.json + artifacts/*.full.json   │
└──────────────────────┬─────────────────────────────┘
                       │ 压缩
┌──────────────────────▼─────────────────────────────┐
│                  紧凑数据层 (LLM 侧)                  │
│  - wallet_summary (0.3 KB)                           │
│  - holdings Top 5 (0.5 KB)                           │
│  - recent_activity Top 8 (1.0 KB)                    │
│  - derived_stats + M2 统计 (0.5 KB)                  │
│  - market_context 摘要 (0.9 KB)                      │
│  - signal_context 摘要 (0.5 KB)                      │
│  - token_snapshots Top 4 (0.6 KB)                    │
│  - signals Top 5 (0.7 KB)                            │
│  ──────────────────────────────                      │
│  总计: ≤ 6 KB (≈ 2000 tokens)                        │
│                                                      │
│  用途：Pi Reflection compact_input                    │
│  约束：Pi maxTokens(output)=3000, input 安全区 <4000  │
└──────────────────────────────────────────────────────┘
```

**边界说明**：
- 上面两层都属于**数据平面**，统一只消费 AVE。
- `onchainos CLI` 不进入完整数据层和紧凑数据层，不向 LLM 回灌 OKX 数据。
- `onchainos CLI` 只存在于**执行平面**：wallet login / security scan / swap / gateway simulate / broadcast。

### 4.2 上下文预算硬约束

**规则**：任何新增字段在合并到 `compact_input` 之前，必须经过以下检查：

```python
def validate_compact_input_size(compact_input: dict, max_bytes: int = 6144) -> bool:
    size = len(json.dumps(compact_input, ensure_ascii=False).encode("utf-8"))
    if size > max_bytes:
        raise ValueError(
            f"compact_input size {size} bytes exceeds budget {max_bytes} bytes. "
            f"Reduce field count or truncate further."
        )
    return True
```

## 五、`distill_wallet_style` 改造后的主流程

```python
def distill_wallet_style(self, *, wallet, chain=None, skill_name=None, extractor_prompt=None):
    # ── Phase 1: 数据采集（M1，并行化）──────────────────
    raw_wallet_profile = self.provider.inspect_wallet(...)
    focus_tokens = _pick_focus_tokens(raw_wallet_profile)
    
    # 并行拉取
    with ThreadPoolExecutor(max_workers=6) as pool:
        token_futures = {pool.submit(self.provider.inspect_token, ...): ref for ref in focus_tokens}
        market_futures = {pool.submit(self.provider.inspect_market, ...): ref for ref in focus_tokens}
        signal_future = pool.submit(self.provider.review_signals, ...)
        macro_future = pool.submit(fetch_macro_context, self.provider)
        # 历史交易分页
        history_future = pool.submit(fetch_full_history, self.provider, wallet, chain)
    
    token_profiles = [f.result() for f in token_futures]
    market_data = [f.result() for f in market_futures]
    signals = signal_future.result()
    macro_raw = macro_future.result()
    full_history = history_future.result()
    
    # ── Phase 2: 预处理（M2 + M3 + M4 并行）────────────
    with ThreadPoolExecutor(max_workers=3) as pool:
        m2_future = pool.submit(pair_and_compute_stats, full_history)
        m3_future = pool.submit(compute_market_contexts, market_data, macro_raw)
        m4_filter_future = pool.submit(build_risk_filters, token_profiles)
    
    trade_stats, completed_trades = m2_future.result()
    market_contexts, macro_context = m3_future.result()
    risk_filters = m4_filter_future.result()
    
    # M4 入场因子（依赖 M2 + M3）
    entry_factors = distill_entry_factors(completed_trades, market_contexts)
    
    # ── Phase 3: 组装 compact_input ─────────────────────
    preprocessed = _preprocess_wallet_data(
        wallet, chain, raw_wallet_profile, token_profiles, signals,
        focus_tokens=focus_tokens,
        trade_statistics=trade_stats,
        market_contexts=market_contexts,
        macro_context=macro_context,
        entry_factors=entry_factors,
        risk_filters=risk_filters,
    )
    
    # ── Phase 4: LLM 蒸馏（M5）────────────────────────
    reflection_spec = self._build_reflection_spec(...)
    reflection_result = self.reflection_service.run(reflection_spec)
    profile, review, strategy, execution_intent = parse_enhanced_output(reflection_result)
    
    # ── Phase 5: Skill 生成 + 验证（M5 + M6）──────────
    # ... compile → validate → promote（保持现有流程）...
    
    # ── Phase 6: 回测与置信度（M6）─────────────────────
    backtest_result = run_backtest(strategy, completed_trades, market_contexts)

    # ── Phase 7: 执行就绪度（执行平面）─────────────────
    execution_readiness = validate_execution_readiness(
        execution_intent,   # M5 生成
        trade_plan,         # skill 生成的计划
    )
    
    # 将置信度写入最终结果
    result["confidence"] = backtest_result.confidence_score
    result["backtest"] = backtest_result.to_dict()
    result["execution_readiness"] = execution_readiness
    
    return result
```

## 八、关键设计决策记录

| 编号 | 决策 | 选择 | 理由 | 可回滚 |
|------|------|------|------|--------|
| D1 | 并行化方案 | ThreadPoolExecutor | AVE Provider 是阻塞 subprocess，asyncio 无法直接加速 | 是 |
| D2 | 市场数据给 LLM 的方式 | 预计算摘要指标 | 原始 K 线太大（16KB+），会导致上下文爆炸 | 是 |
| D3 | compact_input 上限 | 6 KB | Pi maxTokens=3000(output)，input 安全区约 4000 tokens ≈ 6KB JSON | 可调 |
| D4 | 买卖配对算法 | FIFO | 简单可靠，链上交易天然有时间序，FIFO 是最直觉的匹配方式 | 可换 LIFO |
| D5 | 入场因子蒸馏 | 频率统计而非回归 | 样本量 <20 时回归无统计显著性 | 是 |
| D6 | 回测方式 | 信号回放 | 不构建完整回测框架，基于已配对的交易做信号吻合度检查 | 可升级 |
| D7 | 条件评估器 | 安全表达式解析 | 禁止 eval()，防止代码注入 | — |
| D8 | 交互协议分布 | 暂时搁置 | harness 已标注可忽略，AVE 数据不支持 | 待 AVE 升级 |
| D9 | 执行适配层 | onchainos CLI | 执行能力成熟，且可与 AVE-only 数据平面解耦 | 可替换 |
| D10 | 数据边界 | AVE-only for read/backtest | 避免双路径导致蒸馏、回测、执行口径漂移 | 不建议回滚 |

## 九、风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| AVE API 限流导致并行失败 | 中 | M1 延迟退化 | max_workers=4，加指数退避重试 |
| Pi LLM 不遵循增强 schema | 中 | M5 输出不完整 | fallback 到 WalletStyleExtractor；分步验证每个字段 |
| compact_input 超过 6KB | 低 | Pi 输出质量下降 | 硬校验 + 自适应截断（先删 signals，再删 token_snapshots） |
| 交易配对不准确 | 中 | M2/M6 统计偏差 | 标注 confidence，数据不足时明确降级为 `insufficient_data` |
| 生成的 primary.py 运行时错误 | 低 | Skill smoke 失败 | 现有 smoke test 已覆盖；增加条件表达式的 try-except |
| 策略过拟合历史数据 | 高 | 实盘表现差 | M6 置信度仅供参考；头寸建议保守；标注"模拟，非投资建议" |
| onchainos CLI 凭证或钱包状态异常 | 中 | 执行层不可用 | 将执行就绪度与策略评分解耦；live execution 前强制 preflight |
| 执行层反向成为第二数据源 | 中 | 蒸馏口径漂移 | 明确禁止把 onchainos market/signal/PnL 回灌到 compact_input |
