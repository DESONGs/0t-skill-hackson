# M1 - 数据采集与标准化模块

## 模块定位

本模块是整个蒸馏链路的**入口层**，负责从 AVE API 拉取原始数据并统一为下游可消费的标准结构。对应 harness 中的「基础数据层 (The Raw History)」。

---

## 现有代码能力评估

### 已实现 ✓

| 能力 | 代码位置 | 评价 |
|------|---------|------|
| 钱包概况拉取 | `providers.py → AveRestProvider.inspect_wallet` | **好**。调用 `wallet-info` + `wallet-tokens` + `address-txs` 三个 CLI 命令，归一化为标准结构 |
| Token 信息拉取 | `providers.py → AveRestProvider.inspect_token` | **好**。含 market_snapshot + risk_snapshot + holder_snapshot |
| 信号拉取 | `providers.py → AveRestProvider.review_signals` | **基础**。仅按链获取 signals，未支持按 token 精准过滤 |
| 数据归一化 | `providers.py → _token_reference / _normalize_wallet_*` | **好**。处理了 AVE API 返回格式不一致的问题 |
| Focus Token 选择 | `service.py → _pick_focus_tokens` | **好**。按交易频次 + 持仓金额加权排序，Top-4 截断 |
| Mock Provider | `providers.py → MockProvider` | **好**。开发/测试时可脱离真实 API |

### 缺失 ✗

| harness 需求 | 缺失描述 | 优先级 |
|-------------|---------|-------|
| Gas 消耗数据 | AVE `address-txs` 返回中可能包含 gas 信息，但 `_normalize_wallet_activity_item` 未提取 | P2 |
| 交互协议分布 | harness 要求识别交易发生在哪个 DEX（Uniswap/Curve/聚合器），当前 activity 归一化丢失了 DEX 来源信息 | P2（harness 已标注"若无暂时忽略"）|
| 并行数据拉取 | `distill_wallet_style` 中所有 API 调用是**串行**的（inspect_wallet → N 次 inspect_token → review_signals），对于 4 个 focus token 延迟约 4×10s | **P0** |
| 历史交易深度 | `address-txs` 当前只拉最近一页，harness 要求「全量交易流水」以支持买卖配对 | **P1** |
| 合约安全特征 | harness 要求 `getcontractsecurity` 分析地址偏好的合约特征，当前只在 token inspect 里拿到 risk_snapshot | P2 |

## 改进方案

### 1. 并行化数据采集（P0）

当前 `distill_wallet_style` 的采集链路：

```
inspect_wallet → [inspect_token × N] → review_signals  （串行，N=4 时约 60s）
```

改为：

```
inspect_wallet ──┬──→ [inspect_token × N 并行] ──→ 汇总
                 └──→ review_signals ──────────────→ 汇总
```

实现方式：使用 `asyncio.gather` 或 `concurrent.futures.ThreadPoolExecutor`。由于 `AveRestProvider` 内部是 `subprocess.run`（阻塞调用），线程池更合适：

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def _fetch_token_profiles_parallel(provider, focus_tokens, max_workers=4):
    profiles = []
    warnings = []
    with ThreadPoolExecutor(max_workers=max_workers) as pool:
        futures = {
            pool.submit(provider.inspect_token, InspectTokenRequest(token_ref=TokenReference.model_validate(ref))): ref
            for ref in focus_tokens
        }
        for future in as_completed(futures):
            try:
                profiles.append(_json_safe(future.result()))
            except Exception as exc:
                warnings.append({"token_ref": futures[future], "error": str(exc)})
    return profiles, warnings
```

**预计收益**：总延迟从 ~60s 降至 ~15s。

### 2. 历史交易分页拉取（P1）

当前 `address-txs` 只拉一页。需要增加分页参数支持：

```python
def inspect_wallet_full_history(self, payload, max_pages=5):
    all_activities = []
    for page in range(1, max_pages + 1):
        raw = self._cli.run_json("address-txs", "--wallet", payload.wallet,
                                  "--chain", chain, "--page-no", str(page))
        activities = _section_list(raw, "result", "items", "txs")
        if not activities:
            break
        all_activities.extend(activities)
    return all_activities
```

**注意**：全量拉取仅用于 M2（交易配对）的预处理，`compact_input` 仍应截断。

## 输入/输出契约

### 输入
- `wallet: str` — 目标钱包地址
- `chain: str | None` — 链标识（可选，从 wallet profile 推断）

### 输出（标准化结构）

```json
{
  "raw_wallet_profile": { "wallet_summary": {}, "holdings": [], "recent_activity": [] },
  "focus_tokens": [{"identifier": "", "chain": "", "token_address": "", "symbol": ""}],
  "token_profiles": [{"identity": {}, "market_snapshot": {}, "risk_snapshot": {}}],
  "signals": {"signals": []},
  "full_activity_history": [],
  "fetch_metadata": {
    "parallel": true,
    "total_latency_ms": 0,
    "token_fetch_count": 0,
    "activity_pages_fetched": 0
  }
}
```

## 风险与注意事项

1. AVE API 限流：并行拉取时需考虑 rate limiting，建议 `max_workers` 不超过 4
2. `full_activity_history` 数据量可能很大，**不应**直接传入 LLM prompt，仅供 M2 统计预处理消费
3. 测试时使用 `MockProvider`，生产切换需确保 `.env` 中 `AVE_API_KEY` 和 `AVE_DATA_PROVIDER=ave_rest` 配置正确
